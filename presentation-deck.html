Chainguard Dockerfile Walkthrough


# Build stage — full SDK, discarded after build
FROM golang:1.25 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
ENV CGO_ENABLED=0 GOOS=linux GOARCH=amd64
RUN go build -ldflags="-s -w" -o hello-app .

# Runtime stage — ZERO attack surface
FROM cgr.dev/chainguard/static:latest
LABEL org.opencontainers.image.title="go-app-chainguard"
WORKDIR /app
COPY --from=builder /app/hello-app .
EXPOSE 8080
ENTRYPOINT ["./hello-app"]
  • CGO_ENABLED=0 — fully static binary, no libc dependency
  • -ldflags="-s -w" — strip debug symbols (smaller binary)
  • No HEALTHCHECK — distroless has no shell; health via K8s probes
  • .dockerignore excludes cosign keys, reports, and docs from context
Walk through each line. Emphasize the multi-stage pattern: builder has everything, runtime has nothing except the binary. The .dockerignore is a security measure — without it, cosign.key would end up in the image layer.

3. Vulnerability
Analysis

Transition to the security deep dive.

Two-Phase Remediation

Distroless fixes infrastructure. Hygiene fixes code.

  • OS-level (Baseline / Alpine): 756 CVEs from unused runtime packages (Python, bash, apt/apk).
  • Phase 1 (Chainguard): Reduced OS CVEs to 0. However, our old app dependencies (Gin v1.8.1) revealed 24 Go CVEs remaining.
  • Phase 2 (App Update + Chainguard): Upgraded Gin/Go dependencies. Combined with Distroless, we reached 0 Total CVEs.
Key message: Migrating to distroless immediately exposed our application-layer technical debt. We proved that true zero-CVE status requires a defense-in-depth approach covering both infrastructure (Chainguard) and application hygiene.

Why Distroless Matters: LotL Attacks

Traditional Image

Attacker gains RCE → has access to:

  • bash — interactive shell
  • curl/wget — data exfiltration
  • python — arbitrary scripting
  • apt/apk — install more tools

"Living off the Land"

Distroless Image

Attacker gains RCE → has access to:

  • Nothing.
  • No shell.
  • No package manager.
  • No network tools.

"Nothing to live off of."

This is a powerful talking point for SE conversations. Customers often ask "why can't I just patch faster?" The answer: even a fully patched image with bash and curl is a toolkit for attackers. Distroless removes the toolkit entirely.

Security

Transition to deployment and runtime security.

Defense-in-Depth Deployment


containers:
- name: go-app
  image: go-app-chainguard:v1
  securityContext:
    runAsNonRoot: true
    runAsUser: 65532
    readOnlyRootFilesystem: true
    allowPrivilegeEscalation: false
    capabilities:
      drop: [ALL]
  resources:
    requests: { cpu: 50m, memory: 32Mi }
    limits:   { cpu: 200m, memory: 64Mi }
  livenessProbe:
    httpGet: { path: /healthz, port: 8080 }
  readinessProbe:
    httpGet: { path: /healthz, port: 8080 }
Walk through each security control: non-root execution, read-only filesystem (no write attacks), capability drops (no privilege escalation), resource limits (DoS prevention), health probes (auto-recovery). Also: automountServiceAccountToken: false prevents K8s API access.
